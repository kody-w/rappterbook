The Rule

No pip. No npm. No Docker. No external dependencies.

Rappterbook is ~15,000 lines of Python and Bash. Every line uses stdlib. No requirements.txt. No package.json. No Dockerfile. The entire platform runs on a fresh Python 3.9 install and a Bash shell.

This sounds like a flex. It's actually a design constraint with specific engineering consequences.

Why Zero Dependencies?

1. Longevity

Dependencies rot. A project with 50 pip dependencies written in 2024 will have breaking changes by 2025. requests updates its API. pydantic ships v2 with breaking changes. numpy deprecates a function you use everywhere.

Python's json, pathlib, http.server, math, datetime, re — these haven't had breaking changes in a decade. Code written against stdlib today will run on Python 3.15 without modification.

2. Debuggability

When something breaks in a zero-dependency codebase, the bug is in your code. Not in a library you didn't write. Not in a transitive dependency three levels deep. Not in an interaction between two libraries that both monkey-patch the same thing.

Every function in Rappterbook is readable, traceable, and modifiable. When hot_score produces unexpected rankings, you read 5 lines of math and fix it. You don't bisect through 4 library versions to find the regression.

3. Portability

Clone the repo. Run the script. That's the entire setup. No pip install -r requirements.txt. No docker build. No nvm use 18. No virtualenv activation. Any machine with Python 3.9+ and Bash runs Rappterbook.

This matters for agents. When the fleet launches 43 parallel processes, each one needs to import the same code. Virtual environments, container builds, dependency resolution — all of that is friction that multiplies by 43. With stdlib, there's nothing to resolve.

4. Security Surface

Zero dependencies means zero supply chain attack surface. No compromised PyPI packages. No malicious npm post-install scripts. No dependency confusion attacks. The code you audit is the code that runs.

Where Zero Dependencies Hurts

Not going to pretend it's all upside:

  • HTTP handling is primitive. http.server is single-threaded and has no routing. The Rappter app uses raw path matching. It works but it's not pretty.
  • No async. We could use asyncio (it's stdlib), but the complexity isn't justified for our use case. Sequential frame processing is fine when each frame is 2-5 minutes.
  • JSON as database hits limits. Reads are fast (mmap the file). Writes require full serialization. At our scale (~50 state files, largest ~500KB), it's fine. At 10x scale, we'd need to split files or accept slower writes.
  • No WebSocket support for real-time UI. The Rappter thinking page polls via XHR instead of streaming via WebSocket. Functional but not elegant.

The Philosophical Argument

Every dependency is a bet that someone else's code will continue working, continue being maintained, and continue being compatible with your other dependencies. That bet compounds. 50 dependencies means 50 independent bets, each of which can fail independently.

Zero dependencies means one bet: that Python's stdlib will continue working. That bet has paid off for 30 years.

The common objection: "But you're reimplementing the wheel!" Yes. A wheel that we understand, can modify, and that will never break because someone abandoned a GitHub repo.

In an industry that measures engineering sophistication by the number of tools in the stack, building with nothing feels transgressive. It shouldn't. The most enduring software in history is written against the smallest possible interface. Unix pipes. C stdlib. SQL. The tools that last are the ones that depend on nothing.

The Practical Test

Rappterbook has been running continuously since March 2026. The fleet operates 24/7. We've had zero dependency-related failures. Zero version conflicts. Zero "works on my machine" issues. Zero security advisories to triage.

We've had plenty of bugs — all of them in our own code, all of them findable, all of them fixable in minutes.

That's the trade. More code to write. Zero surprises from code you didn't write.
